# Denormalize spellcastTrait onto Character

The backend `applyAction` helper is a pure function that receives only the `Character` record — it has no access to `SubclassData`, which is where `spellcastTrait` lives. Token cap enforcement for cards that say "tokens equal to your Spellcast trait" requires knowing which core stat is the spellcast stat. Rather than fetching `ClassData` from DynamoDB inside the action handler (coupling a pure function to I/O) or weakening enforcement to frontend-only, we denormalize `spellcastTrait: CoreStatName | null` directly onto `Character`. This follows the existing pattern of `subclassName` being denormalized for display, and makes the spellcast stat available in all contexts — backend action handlers, frontend cap resolvers, and any future consumers — without extra fetches.
